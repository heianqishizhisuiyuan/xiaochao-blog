## 先把问题说清：为什么“输出 JSON”这么难
很多团队第一次把大模型接进系统时，都会遇到同一个坑：提示词里说“请用 JSON 返回”，模型也确实“看起来像 JSON”，但到线上就开始出事故：

- 少了必填字段
- 枚举值写错（比如 `status: "ok"` 变成 `status: "okay"`）
- 数字/布尔被写成字符串
- 多输出一段解释文字，导致解析失败
- JSON 结构没错，但语义上不符合约束（例如 `age` 允许 0–120，却给出 999）

本质原因是：纯文本生成没有“语法边界”，而工程系统需要“可验证的边界”。所以这篇文章只讨论一件事：如何把模型输出的自由度收紧到“只可能产出你要的结构”。

## 原理讲透：三条路线，收紧“输出空间”
把 JSON 输出“锁死”，常见有三种路线。它们的核心目标相同：减少模型可选 token 的集合，让不合法的序列根本生成不出来。

### 路线 A：JSON 模式（只保证是 JSON）
流程可以概括为：

- 你告诉模型：输出必须是一个 JSON 对象
- 服务端/SDK确保模型最后给你的内容是“可解析 JSON”

它能解决“多输出一段自然语言”这种低级问题，但对“必须包含哪些键、类型是什么、枚举取值范围”等约束无能为力。

### 路线 B：结构化输出（JSON Schema 严格遵循）
结构化输出的思路是：

- 你提供一个 JSON Schema（描述对象结构、字段类型、必填项、枚举等）
- 通过 strict 模式让模型的输出严格符合 schema
- 你拿到的内容天然可解析且字段齐全（否则会触发拒绝/重试策略）

在工程上，这相当于把“格式校验 + 重试逻辑”从提示词里移到了协议层。

### 路线 C：语法/文法约束（Grammar Constrained Decoding）
如果你在本地推理或某些支持语法约束的运行时（例如 llama.cpp 的 GBNF），还可以更硬核：

- 先用文法描述“允许的输出语言”
- 解码时每一步都只允许符合文法的 token
- 结果是：不合法输出在生成阶段就被禁止

这条路线的优点是“约束更底层、更强”，缺点是“文法/运行时成本更高”。

## 一套可落地的工程流程（推荐按这个顺序）
下面是一套我在项目里更偏向的组合拳：从“能跑”到“能稳”。

1) 先定义业务对象
- 明确你到底要什么：字段名、类型、必填、允许为空、枚举范围、数值区间。

2) 用 JSON Schema 描述它
- 强约束：`required`、`enum`、`additionalProperties: false`
- 能用枚举就别用自然语言

3) 选择输出通道
- 需要调用工具/函数：走函数调用 + `strict: true`
- 只需要结构化回复：走 response_format 的 json_schema（同样 strict）

4) 追加“语义校验”
结构化输出解决的是“结构正确”，不自动保证“业务语义正确”。你仍然需要：
- 数值范围、跨字段约束（例如 `start <= end`）
- 与数据库/外部状态一致性

5) 失败策略设计
- 结构不合法：直接重试（通常是最便宜的）
- 语义不合法：返回错误原因给模型，让它在同一 schema 下修正
- 连续失败：降级到人工/规则引擎或输出“拒绝原因”

## 优点
- 可靠性大幅提升：字段缺失、类型错误、枚举漂移显著减少
- 系统更好维护：schema 就是“契约”，接口变更有据可查
- 上下游更省心：前端/后端/数据层都能基于 schema 做校验与自动生成
- 错误更可观测：你能区分“结构失败”和“语义失败”，从而精准优化

## 缺点
- Schema 设计成本：第一次写好 schema 需要时间
- 不是万能：
  - schema 不解决事实真伪
  - schema 不解决跨请求的长期一致性
- 约束过强会降低表达能力：字段设计不合理时，模型“没法说人话”
- 不同平台支持程度不一：有些环境只能做到“JSON 模式”，做不到严格 schema

## 适用场景
- 需要写入数据库、发起交易、生成配置等“机器要消费”的输出
- 有明确字段结构的场景：表单解析、信息抽取、分类标注
- Agent 工具调用：工具参数必须严格结构化，避免误调用
- 批处理/流水线：需要规模化跑，不能靠人工兜底

## 不适用场景
- 纯内容创作：你希望模型自由发挥、语言更自然
- 需求本身模糊：字段定义不清，强行 schema 只会让系统更脆
- 输出结构经常变：频繁改 schema 会让上下游迭代成本升高

## 常见坑
- 只追求“能 parse”，忽略“additionalProperties: false”
  - 结果是模型塞进一堆你没用的字段，造成误读或污染

- 枚举没写全或写太宽
  - 枚举太宽：等于没约束
  - 枚举太窄：真实业务状态被迫“硬凑”，导致语义错误

- 把长文本塞进结构化字段
  - 结构化输出适合短字段；长文本建议拆分或另走自然语言通道

- 忘记做语义校验
  - “结构正确”不等于“可执行”。尤其是金额、日期、权限等敏感字段

- 重试策略不区分错误类型
  - 结构错误：提示词/模式问题
  - 语义错误：需要把失败原因反馈给模型，让它修正

## 学习路径
1) 先掌握 JSON Schema 的基础关键词
- `type`、`properties`、`required`、`enum`、`additionalProperties`

2) 再理解工具调用/函数调用的多轮流程
- 让模型先“提请求”，你的系统执行，再把结果回传给模型

3) 在项目里把 schema 当作契约管理
- 版本化（例如给 schema 命名与变更记录）
- 做回归测试（用历史输入跑一遍，确保不破坏兼容性）

4) 需要更强确定性时，再研究语法约束
- 适合本地推理、边缘设备或对“输出必须合法”要求极高的系统
