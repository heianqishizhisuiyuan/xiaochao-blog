# RAG 检索增强生成讲透：索引、召回、重排与生成的工程闭环

很多“看起来很聪明”的大模型应用，真正上线后会暴露一个硬伤：

- 没有证据就会编
- 有证据也可能找不到
- 找到了证据也可能拼错、塞太多、回答跑题

RAG（Retrieval-Augmented Generation，检索增强生成）就是把“答案”这件事拆成可控的工程链路：先检索证据，再基于证据生成。

## 原理讲透：把 RAG 当成一条可回归的流水线

如果你只记住一句话：RAG 不是“加个向量库”，而是一个从数据到答案的闭环。

常见的端到端架构可以按模块拆开：

- 数据准备：采集 → 清洗 → 切分（chunk）→ 打标签（metadata）
- 建索引：embedding → 向量索引（vector index）→ 可选的关键词索引（BM25）
- 召回：根据 query 找候选文档（top-k）
- 重排：用更强的模型/规则把候选重新排序（rerank）
- 上下文拼装：去重、压缩、加引用、控制 token
- 生成：让模型“只基于证据”回答，并输出可读结构
- 治理：日志、评测集、回归测试、监控与兜底

## 原理讲透：一次请求的流程（按步骤落地）

把一次请求写成可执行的流程，RAG 才会稳：

1) 规范化 query：统一语言/格式，补齐必要上下文（用户角色、时间范围、产品版本）
2) 召回（retrieval）：
- 向量召回：用 embedding 做语义相似
- 关键词召回：用 BM25 解决专有名词、编号、精确匹配
- 混合召回：两者合并并去重（常见更稳）
3) 重排（rerank）：用 cross-encoder 或更强的模型对候选重新打分
4) 上下文拼装（context building）：
- 去掉重复段落
- 把长文压缩成“可引用证据”
- 控制总长度（给生成留空间）
5) 生成（generation）：明确约束：
- 只能基于给定证据回答
- 给出不确定性声明
- 输出引用来源（例如列出段落标题或文档 id）
6) 记录与回归：把失败样本（找不到、找错、答非所问）沉淀进评测集

## 优点：为什么 RAG 是“上线友好”的路线

- 可控：证据可追溯，能解释“答案从哪来”
- 可更新：内容变更只需更新索引，不必重新训练模型
- 可治理：能做权限过滤（只检索允许的文档）、能做审计
- 成本友好：比直接把全量文档塞进上下文更省 token

## 缺点：RAG 解决不了的事（以及它的新坑）

- 召回失败会让后面全失败：检索阶段找不到，生成再强也无用
- 数据质量决定上限：文档混乱、重复、过期会直接污染答案
- 评测成本不可省：没有离线集与回归，就只能靠“体感调参”
- 长文本与复杂推理仍然难：RAG 给证据，不保证你能推得对

## 适用场景：哪些需求一上来就该用 RAG

- 企业/团队知识库问答：制度、流程、接口文档、FAQ
- 代码与技术文档辅助：从 README、变更记录、ADR 中检索证据
- 产品客服/运营：回答需要引用政策条款、价格、时效等
- 内容生成但要求可追溯：报告/周报/方案必须能给出处

## 不适用场景：别硬上 RAG 的情况

- 需要“纯创造”的内容：例如小说、广告创意（检索反而束手束脚）
- 数据源不可控且无法做权限隔离：检索到敏感信息的风险太高
- 强确定性且不可逆操作：例如自动打款、自动改生产配置（应先走审批链）

## 常见坑：RAG 翻车多半是这些原因

- chunk 切得不合理：太大导致噪音，太小导致语义断裂
- metadata 没设计：缺少版本/权限/时间范围，检索容易“拿错时代的答案”
- top-k 盲目加大：看似召回更全，实际把生成挤爆、答案更差
- 不做重排：向量召回的 top-k 相关性波动大，重排能显著提高稳定性
- 不做混合检索：专有名词、编号、报错信息，用 BM25 常常更靠谱
- 没有“无答案”策略：检索不到也硬答，最终把幻觉当事实传播

## 学习路径：按 1 周把 RAG 做到“能上线”的程度

- 第 1 天：理解基本概念（chunk、embedding、top-k、rerank、hallucination）
- 第 2 天：做最小 demo（固定数据集 + 向量召回 + 引用输出）
- 第 3 天：加入混合检索（BM25）与 metadata（版本/权限）
- 第 4 天：加入重排与上下文压缩（去重、摘要、token 控制）
- 第 5 天：建立离线评测集（10-50 个真实问题），做可重复对比
- 第 6 天：加观测与兜底（召回为空、置信度低、超时降级）
- 第 7 天：上线前复盘：权限、审计、数据更新策略、回滚方案
