# Notes / Quick Notes 内容模型草案（2026-03-09）

## 目标
把“最近在看 / 随手记 / 最近想写”从首页上的静态状态块，逐步升级成可长期维护、可扩展、可独立成型的内容类型。

---

## 一、为什么要单独做内容模型
当前这些状态块已经有价值，但如果一直只是手写数组或临时文案，会有几个问题：
- 很难持续更新
- 很难复用到首页、侧边栏、后续独立页面
- 很难区分“短状态”与“正式文章”
- 一旦数量变多，就会重新退化成杂乱文本

所以需要单独给 notes / quick notes 建一个轻量内容模型。

---

## 二、建议的内容类型拆分

### 1. currently-reading
用途：记录最近在看、在回看、在研究什么。

适合内容：
- 某类主题
- 某种方法
- 某组页面/产品/体验方向
- 最近反复思考的输入来源

### 2. quick-notes
用途：记录短想法、碎片判断、还没长成文章的小东西。

适合内容：
- 一句话判断
- 临时想法
- 改站时的瞬时感受
- 后续可扩写的点子

### 3. upcoming-notes
用途：记录已经决定后面要写，但还没正式写出来的题目。

适合内容：
- 未来文章题目
- 想展开的话题
- 已确定方向但未写成正文的主题

### 4. now-watching（建议新增）
用途：记录最近正在盯、在检查、在观察的问题。

适合内容：
- 页面哪里还不够自然
- 哪个体验点还不稳
- 哪些模块看起来完整但未必真有价值

---

## 三、推荐 frontmatter 字段

每条 note 文件建议至少支持：
- `title`
- `summary`
- `type`
- `updatedAt`
- `items`

示例：

```md
---
title: 最近在看
summary: 最近反复回看、会影响判断和页面气质的内容。
type: currently-reading
updatedAt: 2026-03-09
---

- 最近在反复看信息架构、阅读体验和个人博客设计。
- 也在看怎么让一个站点既能长期写内容，又不会越来越重。
```

---

## 四、后续演进方向

### 阶段 1：单文件列表型 notes（当前）
- 一个文件代表一类状态内容
- 里面用 bullet list 记录条目
- 适合快速落地

### 阶段 2：拆成逐条 note 文件
建议后续升级为：
- `content/notes/currently-reading/*.md`
- `content/notes/quick-notes/*.md`
- `content/notes/upcoming-notes/*.md`

这样每条 note 可以拥有独立字段，例如：
- `date`
- `mood`
- `status`
- `source`
- `pinned`

### 阶段 3：独立页面 / 独立流
后面可以扩展：
- `/notes`
- `/notes/quick`
- `/notes/reading`
- 首页只显示最新若干条
- 详情页或列表页显示完整 notes 流

---

## 五、当前建议落地顺序

### P1 当前轮
1. 先把现有状态块都收进 `content/notes/*.md`
2. 数据层统一解析 notes
3. 首页 / 侧边栏从 notes 内容源读取

### P2 下一轮
1. 增加 `type` / `updatedAt` 字段
2. 新增 `now-watching.md`
3. 开始为 notes 设计独立列表页草案

### P3 后续
1. 拆分成逐条 note 文件
2. 让 notes 和 posts 成为两套并行内容系统
3. 支持更短频的表达形式，而不污染正式文章结构

---

## 六、完成定义（DoD）
当满足以下条件，可认为 notes 内容模型进入可持续阶段：
- notes 已不再依赖手写数组
- 首页 / 侧边栏从 notes 内容源读取
- 至少定义清楚 3~4 种 notes 类型
- notes 与正式文章有清晰边界
- 后续可以独立扩展页面，而不需要推翻现有结构
