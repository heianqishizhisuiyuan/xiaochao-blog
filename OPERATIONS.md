# Blog 执行与协作规则（2026-03-09）

## 1. 目标
把博客项目的开发、调研、验证、沉淀流程固定下来，避免靠临时聊天推进。

---

## 2. 默认工作方式
- 能直接做的任务直接做，做完再汇报
- 复杂任务默认拆分，不硬扛单线程
- 先结论，后细节
- 先验证结果，再说完成
- 完成一部分，提交一部分；禁止最后攒成一大坨

---

## 3. 复杂任务自动进入子代理模式

### 3.1 自动触发条件
满足任意 1 条即可进入子代理模式：
- 预计耗时 > 20 分钟
- 需要 research / dev / qa 角色分工
- 任务涉及 3 个以上文件
- 涉及“方案 + 落地 + 验证”三段链路
- 需要并行推进以缩短总时长

### 3.2 默认分工
- **research**：找方案、找参考、做结构建议、输出风险点
- **dev**：做代码、调结构、补文档
- **qa**：验收功能、跑构建、列问题、做回归

### 3.3 当前约束
- 当前环境暂未发现可直接调用的多 subagent 资源（仅检测到 `main`）
- 因此先将“子代理模式”写成**硬性执行制度**
- 一旦可用 agent / ACP harness 接入，优先按此制度直接启用

---

## 4. 子任务模板
每次拆任务必须写清楚：

### 模板
- **目标**：本子任务要产出什么
- **范围**：允许读写哪些路径
- **约束**：不能改什么 / 必须遵守什么规范
- **验收标准**：什么算完成
- **输出位置**：结果写到哪里

### 示例
- 目标：把静态文章改为 Markdown 内容源
- 范围：`blog/src`、`blog/content`、`blog/docs`
- 约束：不改部署配置；不引入重型 CMS
- 验收标准：新增 Markdown 文件后能自动出现在列表和详情页
- 输出位置：代码改动写入仓库，方案说明写入 `blog/docs/content-pipeline.md`

---

## 5. 文档沉淀规则

### 必须沉淀到文档的内容
- 长期策略
- 信息架构决策
- 内容模型
- 发布流程
- 质量标准
- 常见故障与解决方案

### 沉淀位置
- 总体计划：`blog/PLAN.md`
- 执行规则：`blog/OPERATIONS.md`
- 阶段设计：`blog/docs/*.md`
- 当前已落地专题：`blog/docs/ui-debranding-plan.md`
- 当前已落地验收：`blog/docs/ui-debranding-qa.md`
- 长期项目状态：`memory/projects.md`
- 可复用技能：`skills/*`

---

## 6. Skill 沉淀策略

### 升级为 skill 的条件
同类流程满足以下至少 2 条时，升级为 skill：
- 重复执行 ≥ 3 次
- 输入输出固定
- 跨项目可复用
- 依赖清晰、步骤稳定

### 升级顺序
1. 先写进项目文档
2. 再写成标准流程模板
3. 最后沉淀成 skill

### 原则
- 没沉淀成 skill，不代表不规范
- 先把流程写清楚，再考虑封装
- skill 是放大器，不是唯一依赖

---

## 7. 每轮开发的标准流程
1. 明确本轮目标
2. 判断是否触发子代理模式
3. 写子任务模板
4. 执行开发 / 调研 / 验证
5. 跑构建或关键路径验证
6. 更新文档与项目状态
7. 分批提交 Git
8. 记录下一步

---

## 8. 当前建议的优先执行序列
1. UI 去企业化改造方案与页面清单
2. 内容模型文档
3. Markdown 内容接入
4. 列表 / 归档 / 标签自动派生
5. SEO 与站点基础能力
6. 发布前检查与 QA 流程

---

## 9. 风险控制
- 未验证通过，不汇报“已完成”
- 涉及部署/公网配置变更，单独确认
- 遇到结构性重构，先出方案再改
- 涉及大量重复步骤，优先考虑脚本化或 skill 化
