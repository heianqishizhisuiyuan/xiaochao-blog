# Blog 项目计划（2026-03-09）

## 1. 项目目标
把当前博客从“可看的前端原型”推进为“可长期持续运行的个人博客系统”，重点不是一次性冲刺页面，而是建立：

1. 清晰的信息架构
2. 稳定的内容生产与发布链路
3. 可持续维护的工程规范
4. 可复用的任务拆解与协作机制

---

## 2. 当前现状判断

### 已有基础
- 技术栈已确定：Vue 3 + TypeScript + Vite + Element Plus
- 已有页面骨架：首页 / 文章列表 / 文章详情 / 关于 / 归档 / 404
- 已有静态内容数据，可支撑 UI 与信息架构迭代
- 已有本地开发与构建流程

### 当前主要缺口
1. 当前 UI 仍有明显“企业官网/品牌展示页”味道，不够像个人博客
2. 内容源仍偏静态，缺少可持续内容组织方式
3. 缺少明确的产品阶段计划与验收标准
4. 缺少“复杂任务自动进入子代理/分身模式”的项目规则
5. 缺少沉淀文档，后续容易重复决策、重复返工
6. SEO、内容资产、发布流程、质量保障仍未完整闭环

---

## 3. 总体路线图

### Phase 0：治理与规则固化（立即）
目标：先把“怎么做”写清楚，避免继续靠临时记忆推进。

交付物：
- `blog/PLAN.md`：项目总体计划
- `blog/OPERATIONS.md`：执行/协作/提交流程
- `AGENTS.md`：全局任务分发与子代理触发规则
- `HEARTBEAT.md`：博客持续推进检查项

验收标准：
- 后续就算上下文丢失，也能按文档继续推进
- 复杂任务有明确触发条件，不再临时判断

### Phase 1：内容模型、信息架构与 UI 风格重定向
目标：把博客的“内容系统”与“个人表达 UI”一起定下来，而不是继续沿着企业官网风格修补。

任务：
- 明确文章、分类、标签、归档、精选、推荐的内容模型
- 统一文章元数据字段（标题、摘要、日期、标签、分类、阅读时长、封面、SEO 描述）
- 规划内容存储方式（优先 Markdown/本地内容目录）
- 规划首页、列表页、详情页、关于页的信息层级
- 明确 UI 风格调整方向：从“企业展示感”切到“作者型个人博客感”
- 识别当前页面中所有容易产生企业官网感的模块与文案

交付物：
- 内容模型说明文档
- 示例文章元数据规范
- 页面信息架构草案
- UI 风格重定向清单（保留项 / 删除项 / 改写项）
- `blog/docs/ui-debranding-plan.md`：UI 去企业化详细改造清单
- `blog/docs/ui-debranding-qa.md`：UI 去企业化验收记录

验收标准：
- 新文章可按同一规范落地
- 页面结构围绕内容服务，而不是只服务视觉
- 首页、导航、侧栏、卡片、关于页的气质明确偏向个人博客而非企业官网

### Phase 2：内容接入与渲染链路
目标：把静态 demo 内容升级为真正可维护的内容流。

任务：
- 建立 Markdown 内容目录
- 实现文章索引生成机制
- 实现文章详情内容渲染
- 让分类、标签、归档由内容自动派生

交付物：
- 内容目录结构
- 内容加载与索引逻辑
- 至少 10~20 篇真实/半真实文章样本

验收标准：
- 新增文章不需要手改多处代码
- 列表/归档/详情能从统一内容源生成

### Phase 3：阅读体验与站点完成度
目标：把“能用”提升到“愿意长期用”。

任务：
- 优化文章页排版、目录、上下篇、相关文章
- 优化列表筛选、搜索、空状态
- 完善 About / 首页个人表达
- 增加站点基础能力：404、SEO meta、sitemap、RSS、favicon、分享信息

交付物：
- 阅读体验优化后的页面
- SEO/站点基础文件
- 基础体验测试记录

验收标准：
- 页面阅读负担明显下降
- 搜索引擎与分享场景具备基本可用性

### Phase 4：部署、校验与运营闭环
目标：让博客进入稳定更新状态。

任务：
- 检查 GitHub Actions → Cloudflare Pages 发布链路
- 增加构建前校验（类型、构建、内容格式校验）
- 形成内容发布 checklist
- 形成阶段性优化清单

交付物：
- 发布流程文档
- 发布前检查清单
- 持续优化 backlog

验收标准：
- 发一篇文章有标准流程可走
- 页面、内容、构建、部署至少有基础防呆机制

## 4. 子代理 / 分身协作策略

## 4.1 触发条件（满足任一条即启用）
1. 单任务预计超过 20 分钟
2. 任务可拆为 research / dev / qa 三类并行工作
3. 涉及跨文件、跨模块、跨文档同步
4. 需要先调研再改造再验证
5. 需要产出方案文档 + 代码改造 + 测试结果

## 4.2 默认拆分方式
- **research 子代理**：调研竞品、内容结构、SEO、实现路线
- **dev 子代理**：负责具体功能开发或结构重构
- **qa 子代理**：负责验收、回归、问题清单整理

## 4.3 执行顺序
- 可并行：research + dev
- 依赖验证：qa 在 dev 结果出来后介入
- 高风险结构调整：research → dev → qa 串行

## 4.4 每个子任务必须包含
1. 目标
2. 范围
3. 约束
4. 验收标准
5. 输出位置

## 4.5 当前环境说明
- 当前已检测到的可用 subagent 列表只有 `main`
- 因此“子代理模式”先作为**项目制度与默认执行策略**写入文档
- 当 ACP/更多 agent 可用时，直接按该规范执行，不需要重新定规则

## 5. 持续运行机制

### 5.1 文档沉淀优先级
优先级从高到低：
1. 规则写入 `AGENTS.md`
2. 项目计划写入 `blog/PLAN.md`
3. 执行细则写入 `blog/OPERATIONS.md`
4. 阶段状态写入 `memory/projects.md`
5. 可复用流程再升级为 skill

### 5.2 什么情况下升级为 skill
满足任意两条即可考虑沉淀为 skill：
- 同类任务重复出现 ≥ 3 次
- 步骤固定且可标准化
- 有明确输入输出结构
- 跨项目可复用

候选 skills：
- blog-content-pipeline：文章内容接入与校验
- blog-release-check：发布前检查
- blog-seo-audit：SEO 巡检
- blog-qa-smoke：博客关键路径验收

### 5.3 如果还没做成 skill，如何保证可持续
- 先写入 `AGENTS.md` / `HEARTBEAT.md` / `blog/OPERATIONS.md`
- 确保触发条件、执行流程、验收方式已经文本化
- 保证换一轮会话也能照着继续做

## 8. 完成定义（DoD）
满足以下条件，才算博客进入“可持续运行”状态：
- 有明确项目计划与执行规则
- 有内容模型与内容目录规范
- 有真实内容接入链路
- 有发布前检查流程
- 有复杂任务进入子代理模式的制度
- 即便不依赖临时上下文，也能继续稳定推进
